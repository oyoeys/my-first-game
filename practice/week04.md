Q.1
이 코드에서 AABB를 화면에 그리는 부분만 설명해줘.
pygame.draw.rect 의 인자가 각각 뭘 의미하는지 포함해줘.

    A.1
    이 코드에서 AABB(빨간색 테두리)를 화면에 그리는 핵심 부분은 바로 이 두 줄입니다.

        ```python
# AABB (빨간색 테두리, 두께 2px)
pygame.draw.rect(screen, RED, fixed_rect, 2)
pygame.draw.rect(screen, RED, moving_rect, 2)
```

    이 함수는 `pygame.draw.rect(surface, color, rect, width)`의 형태로 구성되어 있습니다. 코드에 사용된 인자들을 하나씩 자세히 설명해 드릴게요.

            ### 🔍 `pygame.draw.rect` 인자 설명

* **1번째 인자 (`screen`) - 표면(Surface):**
    * 그림을 그릴 **대상 화면**입니다. 위 코드에서는 `pygame.display.set_mode()`로 만든 창인 `screen`을 지정했습니다.
* **2번째 인자 (`RED`) - 색상(Color):**
    * 그려질 사각형의 **색상**입니다. 코드 상단에서 `RED = (255, 0, 0)`으로 정의한 RGB 값을 사용하여 빨간색을 지정했습니다.
* **3번째 인자 (`fixed_rect`, `moving_rect`) - 사각형 정보(Rect):**
    * 사각형의 **위치(x, y)와 크기(너비, 높이)** 정보를 담고 있는 `pygame.Rect` 객체입니다. 이 객체가 차지하는 영역 자체가 바로 AABB(Axis-Aligned Bounding Box)가 됩니다.
* **4번째 인자 (`2`) - 선 두께(Width):**
    * 사각형 **테두리의 두께**를 픽셀 단위로 지정합니다. 이 인자가 아주 중요합니다! 
    * 만약 이 값을 적지 않거나 `0`으로 설정하면 사각형 내부가 색상으로 꽉 채워집니다(앞서 그린 회색 사각형처럼요). 하지만 `2`처럼 0보다 큰 숫자를 넣으면 내부가 빈 **테두리 선**만 그려지기 때문에 충돌 범위를 나타내는 AABB를 시각적으로 표현하기에 적합합니다.

Q.2
sqrt 없이 구현할 수 있어? 차이가 뭐야

    A.2
    네, 당연히 구현할 수 있습니다! 오히려 실제 게임 개발에서는 `sqrt`(제곱근) 연산을 **사용하지 않는 방식**을 훨씬 더 많이 씁니다.

    ### 🧮 `sqrt` 없이 구현하는 원리

    핵심은 **양변을 제곱(Square)해서 비교**하는 것입니다. 거리를 구할 때 마지막에 루트를 씌우는 대신, '거리의 제곱' 값을 그대로 두고 '반지름 합의 제곱'과 비교하면 결과는 완전히 똑같습니다.

    * **기존 공식 (루트 사용):**
        $$\sqrt{(x_2 - x_1)^2 + (y_2 - y_1)^2} \le r_1 + r_2$$
    * **루트 제거 공식 (양변 제곱):**
        $$(x_2 - x_1)^2 + (y_2 - y_1)^2 \le (r_1 + r_2)^2$$

    ### 💻 코드 변경점

    기존의 `math.hypot()` 부분을 아래와 같이 단순 뺄셈과 제곱 연산으로 바꾸면 됩니다. `import math`도 지우셔도 됩니다.

    ```python
    # 두 중심점 사이의 x, y 거리 차이
    dx = center1[0] - center2[0]
    dy = center1[1] - center2[1]

    # 거리의 제곱 (피타고라스 정리에서 루트만 뺀 것)
    distance_squared = dx**2 + dy**2

    # 거리가 두 원의 반지름 합의 제곱보다 작거나 같으면 충돌
    if distance_squared <= (radius + radius)**2:
        screen.fill(YELLOW)
    else:
        screen.fill(WHITE)
    ```

    ### ⚖️ 두 방식의 차이점

    가장 큰 차이는 **연산 속도(퍼포먼스)**입니다.

    * **컴퓨터의 부담:** 컴퓨터에게 제곱근(`sqrt`) 계산은 단순한 덧셈, 뺄셈, 곱셈보다 훨씬 무겁고 복잡한 작업입니다. 반면 `**2`와 같은 단순 곱셈은 아주 가볍게 처리합니다.
    * **최적화:** 지금처럼 오브젝트가 2개일 때는 속도 차이를 체감할 수 없습니다. 하지만 탄막 슈팅 게임처럼 총알과 적이 수백~수천 개씩 등장해서 매 프레임(초당 60번)마다 충돌을 검사해야 한다면, 루트 연산을 뺄셈과 곱셈으로 대체하는 것만으로도 게임의 렉(프레임 드랍)을 크게 줄일 수 있습니다.

    이 최적화된 연산 방식을 적용해서 전체 코드를 깔끔하게 다시 합쳐 드릴까요?

Q.3
그럼 OBB가 무조건 좋은거 아니야?

    A.3
    정말 예리한 질문입니다! 눈으로 보기엔 빈틈없이 딱 맞아떨어지니 **"정확도만 보면 무조건 OBB가 압승"**인 게 맞습니다. 

    하지만 실제 게임 개발에서 OBB만 쓰지 않고 굳이 AABB를 함께 쓰는 결정적인 이유가 있습니다. 세상에 공짜는 없듯이, 바로 **'컴퓨터의 연산량(계산 비용)'** 때문입니다! 

    두 가지의 장단점을 아주 쉽게 비교해 드릴게요.

    ### ⚡ 1. AABB (축 정렬 바운딩 박스)
    * **장점 (초고속 계산):** 컴퓨터에게 AABB 충돌 계산은 더하기 빼기 수준의 아주 쉬운 암산입니다. `A의 오른쪽 끝 > B의 왼쪽 끝` 같은 단순한 숫자 비교 4번이면 끝납니다. 화면에 총알이 1,000개 날아다녀도 순식간에 계산해 냅니다.
    * **단점 (낮은 정확도):** 아까 보셨듯 여백이 너무 많아서, 닿지도 않았는데 부딪혔다고 우기는(?) 경우가 생깁니다.

    ### 📐 2. OBB (회전 바운딩 박스 / SAT)
    * **장점 (완벽한 정확도):** 오브젝트가 아무리 복잡하게 회전해도 실제 닿는 순간을 칼같이 잡아냅니다.
    * **단점 (느리고 무거운 연산):** 아까 코드에 추가했던 삼각함수(회전), 벡터 내적(투영), 분리축 계산 등 복잡한 수학 공식이 총동원됩니다. 만약 화면에 있는 1,000개의 총알을 전부 OBB로 계산하려고 하면 컴퓨터가 비명을 지르며 게임이 뚝뚝 끊기게 됩니다. (프레임 드랍)

    ---

    ### 💡 결론: 그래서 실무에서는 둘 다 씁니다! (투트랩 전략)

    실제 게임 엔진(유니티, 언리얼 등)은 이 두 가지를 스까서(?) 최고의 효율을 뽑아냅니다. 질문자님께서 방금 경험하신 **"노란색 먼저, 그다음 빨간색"** 순서가 바로 그 정석입니다.

    1. **Broad-phase (대충 훑어보기):** 화면의 모든 오브젝트는 일단 가볍고 빠른 **AABB**로 검사합니다. 저 멀리 떨어져 있는 애들끼리는 굳이 복잡한 계산을 할 필요가 없으니까요.
    2. **Narrow-phase (현미경 검사):** 그러다 AABB끼리 겹치는(노란색 화면) '요주의 인물들'이 발생하면, **그때만 OBB(SAT) 공식을 꺼내서** 진짜로 닿았는지(빨간색 화면) 정밀 검사를 합니다.

    이렇게 하면 컴퓨터의 부담을 확 줄이면서도 완벽한 충돌 판정을 만들어낼 수 있습니다. 알고 보니 AABB도 뒤에서 엄청나게 중요한 역할을 하고 있었던 거죠! 😉
